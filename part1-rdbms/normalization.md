## Anamoly Analysis
- Insert Anamoly: The table structure requires an 'order_id' for every entry. If the company hires a new Sales Representative, their details (name, email, office) cannot be added to the file until they make their first sale. We would be forced to create a "dummy" order or leave the order_id and customer_id columns null, which may violate data integrity rules.  
- Update Anamoly: Row 1 (order_id: ORD1114), office_address is "Mumbai HQ, Nariman Point, Mumbai - 400021". Row 37 (order_id: ORD1176), office_address is "Mumbai HQ, Nariman Pt, Mumbai - 400021". If the office address changes, we would have to update it in every single row associated with SR01. If any row is missed (as seen above), the database becomes inconsistent.
- Delete Anamoly: (order_id: ORD1185): This is the only record containing information about product_id P008 ("Webcam", "Electronics", 2100). If we delete this order ID, we lose all existence of the "Webcam" product from our database, including its price and category.

## Normalization Justification
While a single table is easier to view at a glance, the given dataset demonstrates enough examples why a normalization is not "over-engineering -
- In a flat table, redundant information must be updated in every single row. In this dataset, Sales Rep SR01 (Deepak Joshi) has his office address recorded as "Nariman Point" in row 1, but "Nariman Pt" in row 37. In a normalized system, this address is svaed at one place, preventing such discrepancies and ensuring a single source of truth.
- The flat structure ties product information directly to transactions. For example, the Webcam (P008) appears in only one record (Order ORD1185). If that specific order is deleted or canceled, the system loses all record of that product’s existence, including its category and unit price. Normalization protects master data (Products) from being accidentally wiped out by operational changes (Orders).
- A flat table makes it impossible to store data about entities that haven't interacted yet. You cannot add a new Sales Representative or a new Product to this CSV without also creating a dummy order_id. This forces the database to contain "null" or fake transaction data just to store some basic information.
